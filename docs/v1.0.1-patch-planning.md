# SentinelFlow v1.0.1 Patch Planning

## 1. Purpose

This document defines the planning rules for the SentinelFlow v1.0.1 patch after v1.0.0-rc.1 and Milestone 26 customer rollout preparation.

Core decision:

```text
v1.0.1 should focus on bug fixes, onboarding improvements, documentation corrections, and low-risk stability changes.
```

The patch should preserve the validated v1.0.0 release-candidate behavior while addressing customer blockers, hosted API reliability issues, Salesforce package installation problems, workflow defects, audit/replay gaps, and security or privacy issues discovered during early rollout.

## 2. Patch Scope

Included patch scope:

- Bug fixes for validated customer or internal rollout issues.
- Customer onboarding improvements that reduce setup confusion or support burden.
- Documentation corrections for install, support, troubleshooting, security, privacy, or rollout guidance.
- Low-risk Salesforce package stability changes.
- Low-risk hosted API stability changes.
- Safe monitoring, logging, or validation improvements.
- Regression fixes for approval, execution, replay, audit, or dashboard behavior.
- Security or data privacy fixes that can be validated clearly.

Patch behavior goals:

- Keep the existing human approval model.
- Keep production hosted mode aligned with the validated v1.0.0-rc.1 path.
- Keep high-risk architecture work out of the patch.
- Prefer small, testable changes with clear rollback paths.
- Treat customer trust and operational reliability as the primary release goals.

## 3. Patch Candidate Sources

Patch candidates may come from:

- P0/P1 production issue tracking.
- Customer onboarding findings.
- Support SLA escalations.
- Usage monitoring and adoption metrics.
- Feedback-to-roadmap review.
- Salesforce debug logs.
- `Sentinel_Error_Log__c` records.
- Hosted API logs.
- Hosted API error logs.
- Hosted DB health checks.
- Render uptime or cold-start observations.
- Security review findings.
- Install validation results.
- Regression testing after v1.0.0-rc.1.
- Customer-reported documentation gaps.

Every patch candidate should capture:

- Source.
- Severity.
- Customer or org impact.
- Related Salesforce record ids, if applicable.
- Related hosted API or database evidence, if applicable.
- Reproduction steps.
- Expected result.
- Actual result.
- Proposed fix.
- Validation requirements.
- Rollback note.

## 4. Must-Fix Criteria

The following items are must-fix candidates for v1.0.1 unless already resolved by a hotfix:

- P0/P1 customer blockers.
- Hosted API failures.
- Salesforce package install blockers.
- Callout/authentication failures.
- Approval/execution defects.
- Replay/audit defects.
- Data privacy or security issues.

Must-fix handling rules:

- P0 issues require immediate owner assignment and hotfix consideration.
- P1 issues require owner assignment before the v1.0.1 candidate list is closed.
- Security and data privacy issues require explicit validation evidence and customer communication review.
- Approval bypass, unsafe execution, or missing audit evidence must block patch release until resolved or formally accepted as risk.
- Package install blockers must be validated against a clean target org before release.

## 5. Candidate Improvements

Candidate improvements for v1.0.1:

- Clarify installation and onboarding documentation.
- Improve troubleshooting for hosted API, Remote Site, Named Credential, and API key setup issues.
- Improve customer-facing support notes for Render cold starts and retries.
- Tighten validation steps for hosted API health and hosted DB health.
- Improve error messages that help admins resolve callout or permission issues.
- Add low-risk audit, replay, or logging corrections where evidence is incomplete or confusing.
- Add low-risk Salesforce package metadata corrections discovered during customer onboarding.
- Add low-risk support workflow improvements based on repeated customer questions.
- Correct release notes, install guide, support guide, security documentation, or maintenance references.

Candidate improvement acceptance rules:

- The change must reduce customer friction, support burden, or production risk.
- The change must be small enough to validate within the patch window.
- The change must not require broad architecture changes.
- The change must have a clear rollback path.

## 6. Excluded Changes

The following changes are excluded from v1.0.1 patch scope:

- Major AI architecture change.
- Hosted HYBRID Ollama.
- Agentforce production integration.
- Full autonomous remediation.
- Large object model changes.

Additional exclusions:

- New product areas not required to fix a blocker.
- Broad dashboard redesigns.
- Tenant isolation redesign.
- Large data migration.
- New external integration platform work.
- Major packaging model changes unless required to resolve an install blocker.
- Experimental features that would expand validation risk.

Excluded changes should move to future roadmap planning unless they become required to resolve a P0/P1 blocker.

## 7. Validation Requirements

Minimum validation for v1.0.1:

- Hosted API health check passes.
- Hosted DB health check passes.
- Salesforce package validation passes.
- Clean install or clean validation org check passes for package-impacting changes.
- Test incident can be submitted from Salesforce to hosted API.
- Sentinel Incident write-back succeeds.
- Risk score, policy decision, recommendation, and runbook selection are populated.
- Approval workflow works for approval and rejection paths.
- Approved execution creates the expected safe action.
- Replay timeline includes required audit events.
- Dashboard and Org Health Score continue to load.
- Error logging does not expose secrets.
- Security/privacy fixes include focused validation evidence.
- Documentation changes are reviewed for consistency with release notes and maintenance log.

## 8. Release Notes Requirements

v1.0.1 release notes must include:

- Release version and date.
- Summary of patch purpose.
- Fixed customer blockers.
- Hosted API fixes or reliability notes.
- Salesforce package install or metadata fixes.
- Callout/authentication fixes.
- Approval/execution fixes.
- Replay/audit fixes.
- Security or data privacy fixes.
- Onboarding and documentation improvements.
- Known limitations that remain after the patch.
- Upgrade or installation instructions.
- Validation summary.
- Rollback guidance.
- Support contact or escalation path.

Release notes must not promise excluded roadmap work as part of v1.0.1.

## 9. Rollback Plan

Rollback requirements:

- Keep the last known good v1.0.0-rc.1 source state identifiable.
- Keep the last known good Salesforce package manifest available.
- Keep hosted API environment configuration documented.
- Keep database-impacting changes reversible or explicitly documented as non-reversible.
- Preserve Remote Site and Named Credential fallback notes.
- Document any customer-facing workaround used during rollback.

Rollback paths:

- Documentation-only issue: revert the affected documentation file and maintenance entry.
- Salesforce metadata issue: redeploy the last known good package manifest and confirm permission sets, custom metadata, and callout configuration.
- Hosted API issue: redeploy the last known good hosted API version and verify `/` and `/api/health/db`.
- Authentication issue: restore the last known good shared-secret and callout configuration.
- Approval/execution issue: disable execution guidance if needed, keep human review active, and restore the last known good Apex/LWC behavior.
- Replay/audit issue: preserve existing records, restore logging behavior, and document any audit gap.

## 10. Patch Exit Criteria

v1.0.1 may exit planning and move to release execution when:

- All accepted must-fix items have owners.
- All P0/P1 blockers are fixed, hotfixed, or formally accepted as risk.
- Patch scope excludes major architecture and roadmap changes.
- Candidate improvements are limited to low-risk stability, onboarding, and documentation work.
- Validation requirements are defined for every accepted candidate.
- Rollback notes exist for every accepted candidate.
- Release notes owner is assigned.
- Customer communication needs are identified.
- Salesforce package validation plan is ready.
- Hosted API validation plan is ready.
- Maintenance log is updated.

Milestone result:

```text
Milestone 26 - Complete
Next - Milestone 27 Real Customer Pilot
```
